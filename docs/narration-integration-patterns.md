# ナレーター連携の実装パターン

このメモは、Slay the Spire AI Player で改善したナレーター連携を、他のゲーム自動プレイシステムへ移植するための要点です。

目的は、すべての操作を読み上げることではありません。ゲーム進行を止めすぎず、視聴者に判断理由、迷い、リスク、方針が見える場面だけを実況します。

## 基本方針

- ゲーム操作と実況生成は分離する。
- 実況は `say` と `silent` を明示的に選ばせる。
- 低価値な進行、直前と同じ意味の発話、忙しい場面の細かい遷移は読ませない。
- 重要な判断では、実行前の迷いと実行発話を分けられるようにする。
- 複数の実況キューを出してから、最後に1回だけゲーム操作を実行できるようにする。
- TTSを鳴らさない場合は `narration:say` を送らない。必要なら `narration:suppressed` は監視ログ用にだけ使う。

## 推奨アーキテクチャ

```text
Game state
  -> legal action builder
  -> decision model / local policy
  -> command
  -> narration director
  -> optional narration:say sequence
  -> game command execution
```

重要なのは、モデルに直接ゲームコマンド文字列を自由生成させないことです。ゲーム側コードが合法手を作り、モデルには `action_id` を選ばせます。実況文は同じ判断レスポンスに含めてもよいですが、最終的な送信可否はローカルの `NarrationDirector` で再判定します。

## モデル出力

判断モデルに実況も出させる場合は、ゲーム判断と一緒に次のフィールドを返させます。

```json
{
  "action_id": "play_2_0",
  "narration_mode": "say",
  "narration_text": "ここで落とせると次がかなり楽です。",
  "narration_emotion": "thinking",
  "narration_thought": "敵を減らして次ターンの被弾を抑える判断です。"
}
```

`narration_mode` は `say` または `silent` です。`silent` のときはTTSを発生させません。`narration_thought` はUIログ用の短い公開メモで、読み上げ対象ではありません。

## NarrationDirector の役割

モデルが `say` を返しても、そのまま必ず喋らせない方が安定します。Slay the Spire AI Player では、`NarrationDirector` が次の判定をしています。

- 直近と同じ文章、同じ出だし、同じ意味の繰り返しを避ける。
- 重要度が低い遷移は抑制する。
- `silent` 指定を尊重する。
- 場面に応じてローカルの候補文を補う。
- 必要なら1操作に対して複数の発話を作る。

例として、カード報酬では次のような2段階発話が有効です。

```text
うーん、ここは悩みますね。火力札と守り札で見比べます。
今回はポンメルを取ります。序盤戦を長引かせない狙いです。
```

一方で、一本道のルート選択では2段階にすると重複しやすいため、1発話だけにします。

```text
分岐は一本道です。次はイベントで、ここから展開を見ます。
```

このように、複数発話は「迷い」と「決定」が別の情報を持つ場面だけに限定します。

## 場面コンテキストを作る

実況の品質は、モデルに渡すコンテキストよりも、ローカルで抽出した場面情報に強く依存します。

最低限あるとよい情報:

- 現在の画面種別
- 選択肢名
- 選んだ選択肢
- 敵の残り体力、攻撃予定、合計被ダメージ
- 自分のHP、ブロック、エネルギー
- デッキ方針
- 直近の発話履歴
- 視聴者に相談しやすい論点

Slay the Spire AI Player では、カード報酬、イベント、マップ、休憩所、戦闘、勝利後報酬でそれぞれ専用のコンテキストを作っています。他ゲームでも「画面タイプごとの実況材料」を先に構造化すると、定型文だけになりにくくなります。

## 発話スタイル

ナレーター runtime へ送る `narration:say` には、本文以外に話し方の制御も含めます。

```json
{
  "type": "narration:say",
  "text": "被弾は残りますが、先に倒す判断です。",
  "emotion": "thinking",
  "pace": "normal",
  "intensity": "normal",
  "priority": 3,
  "queuePolicy": "enqueue",
  "maxQueueMs": 5000,
  "subtitleOnly": false
}
```

使い分けの目安:

- リーサル、危険ターン、ボス戦、ラン開始/終了は高めの `priority`。
- 低価値な遷移は `dropIfBusy` や `silent`。
- 解説を残したいがTTSは不要な検証では `subtitleOnly: true`。
- TTS再生中に字幕が消えないよう、UI側は発話中/合成中は字幕を保持する。

## 重複を避けるルール

定型文感を減らすため、文章そのものだけでなく、意味の近さも見ます。

実装しやすい判定:

- 正規化した本文の直近一致
- 出だしの一致
- 「討伐」「被弾」「視聴者相談」「次ターン予測」などのモチーフ一致
- 直近数件が同じ角度の説明に偏っていないか

避けたい例:

```text
ルートは一本道です。次はイベント、先の分岐まで見て進みます。
分岐は一本道です。次はイベントで、ここから展開を見ます。
```

この場合は、前置き発話を作らず決定発話だけにします。

## ローカライズと安全な本文生成

ゲーム固有名詞は、英語名をそのまま読ませると違和感が出ます。生成側で日本語またはカタカナへ変換します。

必要な処理:

- カード、敵、レリック、イベント名の辞書変換
- 未知の英語や空の括弧を削るサニタイズ
- `null`、`undefined`、内部ID、コマンド構文を読ませない
- 「残り12」ではなく「残り体力12」のように単位を補う
- ゲーム状態と矛盾する断定を避ける

実況文は、モデルの出力をそのまま使わず、最後に必ずサニタイズします。

## ログと検証

改善ループでは、ゲームを数ターンだけでなく、ある程度長く実走して `actions.jsonl` を確認します。

見るべき項目:

- `narration_status`
- `narration_status_reason`
- `narration_sequence`
- 実際の発話本文
- 同一文/近い文の繰り返し
- TTSが不要な場面で `say` が出ていないか
- 重要場面が `silent` になりすぎていないか
- UI側で字幕が発話中に保持されるか

Slay the Spire AI Player では、`actions.jsonl` と `states.jsonl` を突き合わせて、発話が実際のゲーム状態と合っているか確認しました。他ゲームでも、発話ログだけでなくゲーム状態ログを残すことが重要です。

## 他ゲームへ導入するチェックリスト

1. ゲーム状態を構造化して保存する。
2. 合法手リストをコード側で作る。
3. モデルには `action_id` と任意の実況フィールドだけを返させる。
4. `NarrationDirector` 相当のローカル層を置く。
5. `say` / `silent` / 複数発話 / 抑制をローカルで最終判断する。
6. 発話本文をサニタイズし、ゲーム固有名詞を読み上げ向けに変換する。
7. 重要度、キュー方針、字幕のみモードを設定して runtime へ送る。
8. 実走ログを見て、重複、テンポ、事実誤認を繰り返し直す。

## 移植時の最小実装

最初からすべて入れる必要はありません。別ゲームへ最小導入するなら、まず次の4つで十分です。

- `narration_mode: say | silent`
- 直近発話の重複抑制
- 重要場面だけ `narration:say`
- `actions.jsonl` 相当の発話ログ

その後、画面タイプ別の前置き発話、視聴者相談、固有名詞辞書、キュー制御を足していくと安全です。
